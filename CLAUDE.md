## Architecture

**Stack:** React 19 + Vite, D3 v7, TopoJSON (US Atlas), Zustand v5.

**Color palette (`src/styles/theme.js`):** Exports a `T` object imported by `App.jsx`, `USAMap.jsx`, and `StateMap.jsx`. Premium Dark (Deep Slate + Gold) palette:
- `T.bgPrimary #0f1923` — main app background
- `T.bgSecondary #162030` — header, sidebar
- `T.bgTertiary #1c2a3a` — cards, tooltips, toast
- `T.bgMap #f8f4ee` — map panel (warm white)
- `T.gold #c4902a` — primary gold (unchanged)
- `T.goldBright #e0a830` — hover/score/correct gold
- `T.textPrimary #f0ede8` — main text
- `T.textSecondary #8aacc4` — secondary/muted labels (replaces the old green `#6fa37a`)
- `T.textMuted #4a6a84` — very muted text
- `T.countyFill #c8d8c4`, `T.correct #2d6a4f`, `T.wrong #7a2a1a`, `T.hover #a8c4a0`, `T.dropTarget #1a5a8a`, `T.border #4a8a6a`, `T.water #1a3a5c`

**Three-view flow (`view` in quizStore: `'usa' | 'state' | 'about'`):**
- `USAMap` — D3 SVG of all 50 states with a category bar above. Clicking a state calls `setActiveState({ fips, name })` and transitions to `view: 'state'`.
- `StateMap` — the quiz view. Reads `activeState.fips` and `activeCategory` from the store to decide which feature layer to render. (`UtahMap.jsx` is a legacy file — unused, kept for reference only.)
- `AboutPage` — standalone scrollable page with bio, photo grid, social links, and donation cards. No map or sidebar. Photos served from `public/about/` (canyon.jpg, alpine.jpg, hawaii.jpg). A persistent footer nav in `App.jsx` (32px, transparent, all views) toggles between About and the main app.`

**State management (`src/store/quizStore.js`):**
Single Zustand store. Call `useQuizStore()` with no selector (never pass an object-returning selector — Zustand v5's `useSyncExternalStore` backend sees a new object reference every call and causes infinite re-renders). `StateMap` uses local React state for transient things (hover, drag, attempt counts).

**activeCategory flow:**
- Valid values: `'counties' | 'rivers' | 'peaks' | 'parks' | 'cities'`
- Set via `setActiveCategory(cat)` — also resets `correct`, `wrong`, `score`, `streak`.
- USAMap's category bar sets `activeCategory` in the store before navigating to a state.
- `StateMap` reads it to pick which D3 layer to render.

**TopoJSON data (`public/counties-10m.json`):**
Contains all US counties. FIPS codes are 5-digit strings; first 2 digits are the state FIPS. `StateMap` filters with `fips.startsWith(activeState.fips)`. County names come from `feature.properties.name`.

**Static county data (`src/data/utahCounties.js`):**
`{ name, seat, pop, region }` keyed by 5-digit FIPS for Utah's 29 counties. Other states fall back to TopoJSON names with no seat info.

**Feature data (`src/data/features/`):**
All 50 state files auto-generated by `node scripts/fetch-state-data.js`. Each exports named GeoJSON FeatureCollections:
```js
export const rivers = { type: 'FeatureCollection', features: [...] };
export const peaks  = { type: 'FeatureCollection', features: [...] };
export const parks  = { type: 'FeatureCollection', features: [...] };
export const cities = { type: 'FeatureCollection', features: [...] };
```
Every feature requires `properties.name`. Additional per-category properties:
- **Rivers:** geometry is `LineString` (single chain) or `MultiLineString` (multiple disconnected chains — common for rivers that run through a state in non-contiguous segments). `properties.segment_count` is the number of chains. Segments are filtered by midpoint PIP against the state boundary + border-proximity fallback (0.05° tolerance for border rivers like the Mississippi). Short noise chains (< 0.05° total length) are dropped after chaining. Whitelisted rivers (see `src/data/riverWhitelist.js`) bypass both the length filter and the top-10 ranking cut — they are always included regardless of their OSM coverage.
- **Peaks:** `properties.elevation` (ft)
- **Cities:** `properties.pop`
- **Parks:** `properties.type` (NPS unit type), `properties.states` (2-letter codes for all states the unit spans), `properties.boundary` (simplified polygon geometry, ~40 pts/ring — stored for potential outline rendering; not currently used by StateMap)

`src/data/features/index.js` exports `STATE_FEATURES` — a map of FIPS → lazy dynamic import:
```js
export const STATE_FEATURES = {
  '49': () => import('./utah.js'),
  // ... all 50 states
};
```
`StateMap` loads features on demand via `STATE_FEATURES[activeState.fips]()` in a `useEffect`. This code-splits one chunk per state so only the clicked state's data is downloaded. The loaded module is stored in `loadedFeatures` state; `categoryFeatures` is derived in a `useMemo` that also filters rivers shorter than 0.05° total length (with a whitelist bypass) — this prevents degenerate stubs from generating orphaned sidebar chips with no visible map path.

Data sources:
- **Peaks/rivers:** OpenStreetMap Overpass API
- **Parks:** NPS ArcGIS FeatureServer (`NPSParkBoundaries` service, `fBc8EJBxQRMcHlei` org) — NPS-administered units only; filtered to 11 unit types (National Park, Monument, Recreation Area, Seashore, Lakeshore, Parkway, Preserve, Reserve, Historical Park, Battlefield, Memorial, plus National River and Recreation Area). National Historic Sites and National Forests are intentionally excluded.
- **Cities:** Natural Earth ne_10m_populated_places

Multi-state NPS units (e.g. Great Smoky Mountains, Yellowstone) appear in every state their polygon overlaps. State assignment uses polygon vertex sampling + PIP against `us-atlas/states-10m.json` boundaries — NOT the `STATE` field in the NPS data, which only lists one primary state. Each park feature stores `properties.boundary` (simplified polygon, ~40 pts/ring) and `properties.states` (array of 2-letter codes).

**D3 + React pattern:**
The main D3 `useEffect` in `StateMap` re-runs when `[topo, correct, wrong, quizMode, hoveredId, dragId, selectedId, isDragDrop, countyMeta, activeCategory, categoryFeatures, chipItems]` change. The effect calls `svg.selectAll('*').remove()` then redraws from scratch each run. It returns a cleanup that calls `svg.on('.zoom', null)` to remove the zoom handler before the next run.

Stale-closure problem: `selectedId` is in the D3 effect deps, so the effect re-runs whenever selection changes. Click handlers for non-county features (rivers, peaks, parks, cities) are defined *inside* the effect as `handleNonCountyClick(featureName)`, giving them a fresh `selectedId` closure each time. Do not move this function outside the effect.

`everAttemptedRef` tracks first-try attempts via ref to avoid triggering re-renders.

**Peaks zoom (`activeCategory === 'peaks'` only):**
- `d3.zoom()` with `scaleExtent([1, 50])` and `translateExtent([[0,0],[W,H]])` is attached to the SVG element. Max of 50 allows separating tightly clustered peaks (e.g. Smoky Mountains, ~5-mile radius). Zoom is removed (`svg.on('.zoom', null)`) by the D3 effect cleanup on every re-run; other categories never attach it.
- All geographic content (county underlay + peak triangles) lives inside `<g id="zoom-group" clip-path="url(#map-clip)">`. A `<clipPath id="map-clip">` rect matching the viewBox dimensions (W×H) is defined in `<defs>` so geographic content is clipped to the green border at all zoom/pan positions. Background rect, dot grid, and border rect are drawn outside this group and stay fixed.
- Triangle markers are drawn centered at origin (`M 0,-8 L 6.4,4 L -6.4,4 Z`) with `transform="translate(px,py) scale(1/k)"`. The zoom group's scale k and the triangle's 1/k cancel out, keeping markers pixel-fixed at all zoom levels.
- `data-px` / `data-py` are stored on triangle elements so the zoom handler can update transforms without needing projection or datum access.
- `zoomTransformRef` (useRef) stores the current D3 transform. After each D3 effect redraw, `svg.call(zoom.transform, zoomTransformRef.current)` restores the transform so placing a peak doesn't snap the map back to 1x.
- `zoomScale` (useState) drives the Reset button (`zoomScale > 1.1`) and is updated in the zoom handler. Not in the D3 effect deps — zoom changes don't trigger SVG redraws.
- `showHint` (useState) controls the "Scroll to zoom in" overlay. Set to true on entering peaks mode, dismissed on first user scroll (`event.sourceEvent` check) or after 4 s. Programmatic zoom restore (no `sourceEvent`) does not dismiss the hint.
- `zoomRef` (useRef) holds the D3 zoom behavior so the Reset button can call `d3.select(svgRef.current).call(zoomRef.current.transform, d3.zoomIdentity)`.
- Zoom transform and scale reset to identity in a separate useEffect watching `[activeState?.fips, activeCategory]`, which runs before the D3 effect due to effect ordering.

**Two quiz modes (`quizMode` in store):**
- `'drag-drop'` (default) — user drags a name chip from the sidebar and drops it on the map element. The map container has `onDrop`; elements carry `data-id` attributes so `resolveTarget` can find the drop target by `document.elementFromPoint`.
- `'click-id'` — user clicks a chip to select it (`selectedId`), then clicks a map element to identify it. The D3 click handlers read `selectedId` from a fresh closure (stale-closure pattern — see above).

**`@dnd-kit` is installed but not used.** The app uses native HTML5 drag events (`dragstart`, `dragover`, `drop`) exclusively. Do not refactor to use @dnd-kit.

**Click/interaction model:**
- **Counties:** county `<path>` elements have D3 `.on('click')` handlers directly attached; also support drag-and-drop via `onDrop` on the map container using `data-id` attributes. D3 `mouseover`/`mouseout` only set `hoveredId` and manage the hover fill in click-id mode — no tooltip logic in D3. No SVG text labels drawn on any county.
- **Rivers/parks/cities/peaks:** D3 `mouseover`/`mouseout` only set `hoveredId` (for highlight color). No D3 tooltip calls. No static SVG labels drawn after correct placement.
- **Hover tooltips (all categories):** `handleMapMouseMove` on the inner map div uses `document.elementFromPoint` to read `data-id` and checks `correct.has(id)`. Counties: `"Box Elder County · Brigham City"` (or just `"Name County"` if no seat). Peaks: `"Name · 6,621 ft"`. Parks: `"Name · National Park"`. Cities: `"Name · pop. 689,447"`. Rivers: `"Name"`. No tooltip for unplaced features. `handleMapMouseLeave` clears tooltip unconditionally. Both handlers are wired on the inner `position:relative` div for all categories. Tooltip hidden when `dragId` is set.
- **County underlay:** when `activeCategory !== 'counties'`, county outlines render as a faint underlay (`pointer-events: none`) for geographic context.

**Wrong-drop persistence (`dragId` + `keepDragIdRef`):** In drag-drop mode, when a chip is dropped on the wrong target, `dragId` is left set to the dropped chip's id (not cleared). This keeps the chip highlighted in the sidebar (gold border via `held={dragId === id}`) so the user can immediately re-drag it. Implementation: `keepDragIdRef` (useRef) is set to `true` in `handleDrop` on a wrong drop; `handleDragEnd` checks the ref — if true, it clears the ref but leaves `dragId` intact. `dragId` is cleared on: correct drop, dropping on empty space (no `data-id` target), starting any new drag (`handleDragStart` sets `keepDragIdRef=false` then `setDragId(newId)`), clicking map background (not a `[data-id]` element), or quiz reset. Do not use a separate `heldChipId` state — this was tried and removed because `dragend` always fires after `drop`, making it impossible to keep a separate state synchronized.

**SVG border:** Drawn as a `<rect>` inside the SVG viewBox (not CSS border) so it isn't clipped by `overflow: hidden` containers.

**Height chain:** Outer div in `App.jsx` is `height: 100vh`. Children use `flex: 1; min-height: 0` — never use `calc(100vh - Npx)` or the SVG bottom gets clipped.

**Fonts:** Cormorant Garamond (headings) + Raleway (UI), loaded from Google Fonts in `index.html`.

## Regenerating feature data

Run `node scripts/fetch-state-data.js` to re-fetch all 50 states from Overpass + Natural Earth. Pass state names as args to regenerate a subset: `node scripts/fetch-state-data.js Utah California`.

The script overwrites `src/data/features/{state}.js` and rewrites `index.js`. The Overpass endpoint used is `overpass.openstreetmap.fr` — it requires an explicit `User-Agent` and `Accept: application/json` header (the default Node.js fetch User-Agent gets a 403). The main `overpass-api.de` server currently returns 406 from this IP.

Pass `--parks-only` to refresh only the parks export for each state (reads existing rivers/peaks/cities from the current file, replaces parks from NPS, no Overpass calls): `node scripts/fetch-state-data.js --parks-only`

Pass `--rivers-only` to refresh only rivers (reads existing peaks/parks/cities, re-fetches rivers via Overpass): `node scripts/fetch-state-data.js --rivers-only`

Pass `--peaks-only` to refresh only peaks (reads existing rivers/parks/cities, re-fetches peaks via Overpass with PIP validation): `node scripts/fetch-state-data.js --peaks-only`

**Peak PIP validation:** `fetchPeaks` runs every fetched node through `withinFeature([lon, lat], boundary)` using the same us-atlas state polygon used for rivers. Nodes whose OSM coordinates fall outside the state boundary are dropped with a log message. This catches peaks from neighboring states that bleed into the bounding box query (common along state lines — Ohio had many KY/WV nodes; Tennessee had many VA/NC/GA nodes). PIP validation happens inside `fetchPeaks`, so it applies in both full runs and `--peaks-only` runs.

**Peak coordinate overrides (`PEAK_OVERRIDES`):** A `const PEAK_OVERRIDES` map at the top of `fetch-state-data.js` holds hardcoded correct coordinates (keyed by exact OSM `name` tag) for peaks where OSM has misplaced nodes. Overrides are applied before PIP validation. Add entries here when a significant peak fails the PIP check due to an OSM inaccuracy rather than a true location issue. Current overrides: `'Mount le Conte - High Top': [-83.4432, 35.6543]`.

**Park centroid overrides (`PARK_OVERRIDES`):** A `const PARK_OVERRIDES` map at the top of `fetch-state-data.js` holds per-state centroid coordinates for NPS units whose polygon spans multiple disconnected states. The NPS ArcGIS centroid for such units lands outside all actual sites (e.g. Manhattan Project's centroid is in Nebraska). Structure: `{ UNIT_NAME: { STATE_ABBR: [lon, lat] } }`. In `buildNPSParksMap`, when adding a park to a state's list, the state-specific override coordinate is used as the Point geometry instead of the shared centroid. The `boundary` and `states` properties still come from the full NPS polygon. Current overrides: `'Manhattan Project National Historical Park': { TN: [-84.2696, 35.9312], NM: [-106.2951, 35.8801], WA: [-119.5179, 46.5504] }`. The three state data files (tennessee.js, new-mexico.js, washington.js) have been manually patched with these coordinates.

**River segment filtering:** Uses midpoint-based PIP (not endpoint-based). A way segment is kept if its midpoint is inside the state polygon OR within 0.05° of the state border line (for border rivers). After chaining, chains shorter than 0.05° total length are dropped as noise fragments. `CHAIN_TOL = 0.002°` (≈220 m) controls how close endpoints must be to merge. A post-map total-length filter (>= 0.05°) drops any surviving degenerate stubs.

**Spatial coherence filter (`coherentChains`):** Applied after `chainSegments()` to reject geographically anomalous chains caused by OSM ways from a neighboring state bleeding into this state's bounding box. Algorithm: the longest chain is the "anchor"; any other chain is dropped if it is both (a) shorter than 20% of the anchor length AND (b) more than 1.5° from the anchor's geographic center. This correctly removes false-attribution blips (e.g. the Elk River's NE Tennessee segment caused by 51 OSM ways from the VA/NC border area) while preserving legitimate multi-section rivers (e.g. Tennessee River's genuine East TN + West TN sections, which are only 3° apart and both > 40% of anchor). Implemented in `coherentChains(chains)` in `fetch-state-data.js`, called inside `makeRiverFeature`.

**River whitelist (`src/data/riverWhitelist.js`):** `RIVER_WHITELIST` is an array of lowercase base names (e.g. `'jordan'` matches "Jordan River"). `isWhitelisted(name)` is imported by both `fetch-state-data.js` and `StateMap.jsx`. Whitelisted rivers bypass: (1) the per-chain 0.05° MIN_SEG_LEN filter, (2) the post-map total-length filter, and (3) the top-10 ranking slice — they are separated from other rivers before the slice and always included. Add rivers here when they are historically/culturally significant but have short or heavily OSM-fragmented Utah segments (Jordan River has 200+ tiny OSM way segments that don't connect well at CHAIN_TOL=0.002°).

**Alaska antimeridian:** Alaska uses `d3.geoAlbers().rotate([154, 0]).center([-2, 58.5]).parallels([55, 65])` instead of Mercator in `StateMap.jsx` (FIPS `'02'`). Mercator cannot handle Alaska's high-latitude span or Aleutian Islands near the antimeridian. The projection is selected before `fitExtent` so the same projection is used by the county underlay and all feature layers. In `fetch-state-data.js`, Alaska coordinates are normalized (lon > 180 → subtract 360, lon < -180 → add 360) and segments west of -170° are dropped — no named rivers exist in the far Aleutians and they cause distortion.

**Peaks timeout:** The peaks Overpass query uses `[timeout:90]`. Large states (CA, AK, TX, NY, FL, MI, NC, VA) have thousands of OSM peak nodes and silently return 0 results with `[timeout:30]`. If a state shows 0 peaks after a run, increase the timeout first before debugging PIP filters.

**Vite static import analysis:** Vite resolves all dynamic import paths in `index.js` at dev-server startup. If a state file is missing, Vite errors. Write an empty stub (empty FeatureCollections for all four exports) to unblock the server while a full fetch is running.

**FIPS key ordering in `fetch-state-data.js`:** `Object.entries(STATE_FIPS)` returns integer-index keys (`'10'` Delaware through `'56'` Wyoming) before string keys (`'01'` Alabama through `'09'` Connecticut). States are processed in this non-alphabetical order but all 50 are covered.

## Adding enriched county seat data for a new state

County quiz works for any state — TopoJSON has all 50. To add county seat info (like Utah has), create `src/data/<state>Counties.js` with `{ name, seat, pop, region }` keyed by 5-digit FIPS, and add a branch in the `countyMeta` useMemo in `StateMap.jsx`.
